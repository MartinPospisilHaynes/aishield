1. PŘEHLED ARCHITEKTURY
================================================================================

Scanner je 4-stupňový pipeline běžící na VPS (FastAPI backend):

┌─────────────────────────────────────────────────────────────────────────────┐
│  UŽIVATEL zadá URL na aishield.cz                                          │
│                                                                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌───────────┐ │
│  │ 1. WEB       │───>│ 2. DETECTOR  │───>│ 3. CLASSIFIER│───>│ 4. REPORT │ │
│  │    SCANNER   │    │ (signatury + │    │ (Claude AI   │    │ (HTML     │ │
│  │ (Playwright) │    │  heuristiky) │    │  verifikace) │    │  report)  │ │
│  └──────────────┘    └──────────────┘    └──────────────┘    └───────────┘ │
│        │                    │                    │                  │       │
│   Stáhne HTML,        Najde kandidáty      Rozhodne true/     Generuje    │
│   skripty, cookies,   na AI systémy        false positive     compliance  │
│   network requesty,   (raw findings)       pro každý nález    report      │
│   screenshoty                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

Navíc je tu "safety net" — hardcoded seznam nástrojů, které NIKDY nemají
být označeny jako deployed AI (GA4, GTM, Seznam Retargeting, Heureka).
Ty se automaticky přesunou do false-positives bez ohledu na verdikt detectoru
i classifiera.

────────────────────────────────────────────────────────────────────────────────
2.1  PIPELINE (pipeline.py — 225 řádků)
────────────────────────────────────────────────────────────────────────────────

Orchestrátor celého procesu. Řídí pořadí kroků:

  1. Nastaví scan status = "running" v DB
  2. Spustí WebScanner.scan(url) — Playwright crawl
  3. Spustí AIDetector.detect(scanned_page) — hledá AI systémy
  4. Pokud nalezeny findings → spustí AIClassifier.classify(findings, url)
  5. SAFETY NET: projde všechny výsledky a násilně přesune NON_AI_TOOLS
     (GA4, GTM, Seznam Retargeting, Heureka) do false-positives
  6. Uloží findings (deployed) a false_positives do DB (tabulka scan_findings)
  7. Nahraje screenshot do Supabase Storage
  8. Nastaví scan status = "done"

Chybové ošetření:
  - Celý pipeline je v try/except — pokud cokoliv selže, status → "error"
  - Classifier má vlastní try/except — pokud Claude API selže, findings
    se označí jako neklasifikované (ale NE jako deployed=false)

Vrací slovník: {status, scan_id, url, total_findings, false_positives, findings[]}

2.2  WEB SCANNER (web_scanner.py — 209 řádků)
────────────────────────────────────────────────────────────────────────────────

Playwright-based headless browser scan. Klíčová konfigurace:

  Browser:          Chromium (headless, no-sandbox)
  User-Agent:       Chrome 120 na Windows 10
                    "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
                     (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36"
  Viewport:         1920 × 1080
  Locale:           cs-CZ
  Timezone:         Europe/Prague
  SSL chyby:        Ignorovány (ignore_https_errors=True)
  Navigation:       page.goto(url, wait_until="domcontentloaded")
  Čekání:           + 10s manuální čekání na network idle (0 requestů po 500ms)
                    + 5s extra sleep pro dynamický obsah (JS chatboty apod.)
  Timeout:          45s celkem
  
Co extrahuje (dataclass ScannedPage):
  - url:              zpracovaná URL
  - html:             kompletní innerHTML (document.documentElement.innerHTML)
  - html_hash:        SHA-256 hash HTML pro change detection
  - external_scripts: seznam src atributů všech <script src="...">
  - inline_scripts:   obsah všech <script> tagů bez src
  - iframes:          src atributy všech <iframe>
  - meta_tags:        dict name→content všech <meta> tagů
  - cookies:          seznam cookies (name, value, domain, path)
  - console_messages: zachycené console.log/warn/error zprávy
  - network_requests: URL všech HTTP požadavků (zachyceno přes page.on("request"))
  - screenshot_viewport: bytes PNG screenshotu viewportu
  - screenshot_full:     bytes PNG celé stránky (s fallback na viewport)

Technické detaily:
  - Network requesty se zachycují přes callback na page.on("request")
  - Console messages přes page.on("console")
  - Screenshoty: viewport + full_page=True (s fallback pokud full selže)
  - JavaScript evaluace pro cookies, inline scripts, external scripts, meta tags

2.3  DETECTOR (detector.py — 700 řádků)
────────────────────────────────────────────────────────────────────────────────

Dvoustupňová detekce AI systémů:

STUPEŇ 1: SIGNATUROVÝ MATCHING
  
  Prochází databázi ~60 známých AI systémů (signatures.py) a pro každý:
    a) Hledá signatury v HTML (.lower())
    b) Hledá script_patterns v externích skriptech (URL)
    c) Hledá script_patterns v inline skriptech (obsah)
    d) Hledá iframe_patterns v iframe src
    e) Hledá cookie_patterns v názvech cookies
    f) Hledá network_patterns v síťových požadavcích

  Confidence scoring:
    base = 0.3 * počet_signatur_matched + 0.1 * počet_evidencí
    minimum 0.6, maximum 1.0
  
  Evidence se ukládají (typ + matched string) pro transparentnost.

STUPEŇ 2: HEURISTICKÁ DETEKCE (16 metod)
  
  Po signaturovém matchingu se spustí 16 nezávislých heuristických analýz
  (metody 2a až 2p). Každá hledá jiný typ stopy AI na stránce.
  
  Deduplikace: heuristiky přeskočí nálezy, které už byly nalezeny
  signaturovým matchingem (srovnání přes .lower() jména).
  
  Kompletní výčet v sekci 3.

2.4  CLASSIFIER (classifier.py — 296 řádků)
────────────────────────────────────────────────────────────────────────────────

AI verifikační vrstva pomocí Claude (Anthropic API).

  Model:            claude-sonnet-4-20250514
  Max tokens:       4096
  Vstup:            zjednodušený JSON s findings + URL
  
System prompt (zkráceno):
  "Jsi expert na EU AI Act. Pro každý nalezený systém urči:
   - deployed: true/false (je systém skutečně nasazený a funkční?)
   - risk_level: high/limited/minimal
   - ai_act_article: relevantní článek EU AI Act
   - classification_text: popis v češtině
   Tyto nástroje VŽDY označ jako deployed=false:
     Google Tag Manager, Google Analytics 4, Seznam Retargeting, Heureka"

Vstupní formát pro Claude:
  Zjednodušený JSON — pro každý finding posílá:
    name, category, signature_matched, confidence, evidence_types[]

Výstupní formát:
  Claude vrací JSON array:
  [{"name": "...", "deployed": true/false, "risk_level": "...", ...}]
  
  Parser: 
    1. Odstraní markdown code fences (```json ... ```)
    2. json.loads(raw_text)
    3. Mapuje výsledky zpět na findings podle jména

FALLBACK (pokud API selže):
  - Všechny findings se označí jako deployed=true (konzervativní)
  - Loguje se warning
  - Scan pokračuje bez AI verifikace

ZNÁMÝ BUG:
  - JSON parse error (řádek 242) — Claude občas vrátí text místo JSON
  - Tento bug byl zaznamenán v posledním skenu (20:46:27 CET)
  - Při parse chybě: fallback → vše deployed=true

Cenový odhad:
  $0.80/1M vstupních tokenů, $4.00/1M výstupních tokenů
  ~$0.005–0.02 na jeden sken (závisí na počtu findings)

  - Detail každého findings (kategorie, riziko, AI Act článek, akce)

odeslat emailem nebo zobrazit v browseru.

5. ZNÁMÉ PROBLÉMY A PŘÍČINY NEKONZISTENCE
================================================================================

Zjištěný problém: Při testování webu desperados-design.cz scanner v 8 z 10
skenů našel 1–2 AI systémy, ale v jednom skenu našel 11. Proč?

PŘÍČINA 1: DYNAMICKÉ NAČÍTÁNÍ OBSAHU (CRITICAL)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Chatbot skripty (Smartsupp, Tidio, Zendesk, Tawk.to aj.) se často načítají:
  - Podmíněně podle geolokace (IP adresy VPS vs. lokální sken)
  - Podle cookies (první vs. opakovaná návštěva)
  - Podle A/B testů (náhodný bucket)
  - Podle denní doby (pracovní hodiny vs. noc)
  - Lazy loading (po scrollu, po X sekundách, po interakci)
  - Některé chatboty se zobrazí až po kliknutí na trigger button

  Scanner čeká 10s na network idle + 5s extra, ale:
  - Některé skripty se načtou až po 20–30s
  - Lazy-loaded chatboty vyžadují scroll/interakci
  - A/B testy mohou vybrat variantu BEZ chatbotu
  
  DOPAD: Při jednom skenu se stihne načíst více skriptů, při jiném méně.
  To způsobuje obrovskou variabilitu v počtu nálezů.


PŘÍČINA 2: PŘÍLIŠ GENERICKÉ SIGNATURY (HIGH)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Některé signatury jsou tak generické, že matchují i na stránkách,

ZÁVĚR: Z 11 "nasazených AI systémů" jsou pravděpodobně:
  - 0–2 skutečné true positives (Zendesk, možná Meta Pixel)
  - 9–11 false positives způsobené generickými signaturami + classifier failure

Proč občas najde jen 1–2:
  - Claude classifier funguje správně → odfiltruje false-positives
  - Timing je takový, že se nenačtou všechny skripty
  - Oba faktory (classifier OK + méně raw detections) dávají správný výsledek

Proč občas najde 11:
  - Classifier JSON parse error → fallback (vše deployed=true)
  - Timing je takový, že se načte maximum skriptů
  - Generické signatury matchují na běžný obsah
  - Bez classifiera nic neodfiltruje false-positives

APPENDIX C: DATAFLOW DIAGRAM

User → [URL] → FastAPI endpoint → pipeline.run_scan()
  │
  ├─ 1. DB: INSERT scan (status=running)
  │
  ├─ 2. WebScanner.scan(url)
  │     ├─ Playwright: launch browser
  │     ├─ Navigate to URL (domcontentloaded)
  │     ├─ Wait: network idle (10s) + sleep (5s)
  │     ├─ Extract: HTML, scripts, iframes, cookies, network, console
  │     ├─ Screenshots: viewport + full page
  │     └─ Return: ScannedPage dataclass
  │
  ├─ 3. AIDetector.detect(scanned_page)
  │     ├─ Stage 1: Signature matching (60+ signatures × 6 data sources)
  │     ├─ Stage 2: Heuristic detection (16 methods, 2a–2p)
  │     ├─ Deduplication (name.lower())
  │     └─ Return: list[Finding]
  │
  ├─ 4. AIClassifier.classify(findings, url)
  │     ├─ Build Claude prompt (system + user)
  │     ├─ Call Claude API → JSON response
  │     ├─ Parse JSON → deployed=true/false for each finding
  │     ├─ FALLBACK (on error): deployed=true for all
  │     └─ Return: classified findings
  │
  ├─ 5. Safety net: move NON_AI_TOOLS to false_positives
  │     (GA4, GTM, Seznam Retargeting, Heureka)
  │
  ├─ 6. DB: INSERT scan_findings (deployed + false_positives)
  │
  ├─ 7. Supabase Storage: upload screenshot
  │
  └─ 8. DB: UPDATE scan (status=done, results)
